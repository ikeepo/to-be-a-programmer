# Matrix is Important 
## Computer Graphics - Matrix Transformation
如果把计算机图形学（Computer Graphics）、浏览器渲染引擎、游戏引擎乃至 VR/AR 的底层逻辑抽象到极致，它们的核心本质就是一个词：**矩阵变换（Matrix Transformation）**。
# Learning Roadmap
MIT 18.06设计初衷是**传统大学数学课**——侧重于手算方程组 $Ax = b$、求行列式、特征值以及严密的数学证明。
要让线性代数不再成为你前进的障碍，核心在于**从“数学家模式”彻底切换到“工程师模式”**。
How?
### 1. 改变视角：从“公式推导”转向“几何直觉”
3Blue1Brown-Essence of Linear Algebra
### 2. 工程师需要掌握什么
Shape Matching
几何算子的物理语义
空间变换与投影
### 3. 学习路径：用代码（Code-First）代替纸笔
用代码实现公式，而不是手动推导；
代码错误就是最好的学习锚点；
### 4. 把algebra当作API来调用
掌握格式、语义
### 5. 混淆了“数学家”与“工程师”的踏实感
总感觉不会推导就是没学会，没学会就不踏实；
这是基于做题家的规训：题目要推导才算是解出来了，这本质是一种身体trauma；

> [!工程师] 工程师的踏实
> **不来自于推导，而来自于“掌控”**。
> 工程师的学会：
> [边界理解] 知道输入输出是什么，以及它的适用场景与局限 
> ↓
>  [结果预测] 改变输入参数时，能在脑海里准确预测输出的变化 
>  ↓ 
>  [故障诊断] 当系统报错或结果异常时，能定位是哪个算子/维度出了问题


> [!NOTE] Just-In-Time Precision
> 工程师建立直觉 ==> 用直觉解决问题 ==> 用代码验证直觉并建立安全感
> 
> 因为代码介于高语义抽象与低底层复杂度之间。既能实际操作，又不必做太多无效底层工作；
> 
> 这是现代社会必须的Cognitive Offloading技能；
> 
> 只有当高层直觉解释不了现实现象时，才值得花时间去推导底层的数学。
