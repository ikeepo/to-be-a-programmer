# DDD

# DDD价值演变
核心能力：
1. Bounded Context 设计
划分系统边界
防止语义污染
2. Ubiquitous Language
统一业务语言
防止 AI 误解 domain
3. Aggregate Design
控制一致性边界
控制数据依赖关系

# 程序员角色转变
传统程序员：`需求 → 写代码 → 调试 → 交付`

不再需要码农，但是需要更多的架构师。横向扩展懂各个领域底层原理的架构师。
而DDD是其中关键的一个认知，用于框定使AI避免幻觉混乱。

# DDD vs 其他设计模式

首先理解什么是设计模式，它是一种逻辑组织形式，从某一个视角下将某一个主题的所有知识尽可能完备合理的组织在一起。

传统定义是Design Pattern is a reusable solution to a recurring problem.，这是因为英文世界中习惯用**解决方案**视角看问题，而不从**认知角度**。


程序本质是对现实业务的一种建模。

DDD与其他程序设计模式有根本性的区别，DDD在实现对现实重构的同时，对现实进行最原子化的解构（业务角度，DDD关注的是业务语义）。DDD可以解构其他任何的设计模式，但反过来不行。也就是说，DDD是更底层但同时更完备的一种建模思维。

因为DDD的原子化解构建模方式，它可以用于任何事情，一个新学科、一个新工具、一个新工作、一个程序框架、一种编程语言比如CSS、Python、HTML等等。

DDD 更像建筑里的蓝图，而其他设计模式更像施工工艺。但DDD并不是设计模式中的C语言，也就是不能取代其他设计模式。

传统设计模式主要组织的是解决方案（Solution），而DDD首先组织的是概念（Concept）。它通过统一语言、领域模型和边界上下文，把现实世界中的概念、关系和语义抽象成一个可计算的模型，然后再选择合适的实现方式。因此，DDD不仅是一种软件设计方法，更是一种以语义为中心的认知建模方法。这种能力并不限于软件开发，而可以迁移到学习任何具有明确领域知识的学科。
